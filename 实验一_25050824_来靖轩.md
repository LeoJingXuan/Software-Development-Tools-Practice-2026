### 步骤一 安装 Node.js 与 npm

`Node.js` 是整个 Web 开发环境的基石 —— 它既是一个 JavaScript 运行时，也内置了 `npm` 包管理器。

> 💡 **下载**
> 访问 https://nodejs.org/ 下载 **LTS 版本**。

![alt text](image-15.png)
*△ 图 1 · Nodejs官网下载界面*

安装是标准的 Windows Installer 向导流程，但有**两处不能无脑点 "Next"**：

- 在 **Custom Setup** 页面，确保勾选 **"npm package manager"** 和 **"Add to PATH"** —— 前者保证 `npm` 随 Node.js 一同安装，后者让终端随处可敲 `node` 和 `npm`；
- 安装末尾出现的 **"Tools for Native Modules"** 复选框，**务必勾上** —— 自动安装 `windows-build-tools`（Python + C++ 编译链），日后 `node-gyp` 报错时可免去排雷之苦。

### 步骤七 部署到 GitHub Pages

博客目前只在本机 `localhost` 可见。要让世界访问，需要把静态文件托管到公网服务器上。**GitHub Pages** 是最适合静态站点的免费方案。

#### 7.1 创建 GitHub 仓库

1. 登录 [GitHub](https://github.com/)，右上角 **"+"** → **"New repository"**
2. 仓库名**必须**设为 `<你的用户名>.github.io`
3. 可见性选 **Public**，**不要**勾选 *"Add a README file"*
4. 点击 *"Create repository"*

> 📌 **为什么仓库名非此不可？**
> GitHub Pages 域名约定：名为 `<user>.github.io` 的仓库自动映射为 `https://<user>.github.io`。若仓库名随意取，URL 会多出一层路径前缀，导致资源路径错乱。

### 5.3 静态博客 vs 动态博客

| 对比维度 | 静态（Hexo / Hugo / VuePress） | 动态（WordPress / Typecho） |
|:--:|:--:|:--:|
| 页面生成时机 | **构建时**预先生成全部 HTML | **请求时**由服务器实时生成 |
| 数据库 | 不需要 | 需要（MySQL / PostgreSQL） |
| 加载速度 | 极快（纯静态文件，CDN 友好） | 较慢（执行后端代码 + 查数据库） |
| 安全性 | 高（无后端，攻击面极小） | 需持续关注安全更新 |
| 管理后台 | 无（Markdown + 命令行写作） | 有（Web 后台界面） |
| 托管成本 | 极低（GitHub Pages / Vercel 免费） | 需支持 PHP + MySQL 的主机 |
| 适合人群 | 技术博客 / 文档站 / 追求极简 | 企业官网 / 电商 / 非技术人员维护 |

| 工具 | 在 Web 开发中的角色 | 对应的通用概念 |
|:--:|:--:|:--:|
| Node.js | JavaScript 脱离浏览器运行的环境 | **运行时（Runtime）** |
| npm | 依赖声明、安装、版本锁定 | **包管理器（Package Manager）** |
| Git | 代码版本追踪、协作、远程同步 | **版本控制（Version Control）** |
| Hexo CLI | 项目脚手架、构建、本地服务 | **构建工具链（Build Toolchain）** |
| `hexo server` | 开发时提供 HTTP 服务 | **开发服务器（Dev Server）** |
| `_config.yml` | 站点全局参数 | **配置管理（Configuration）** |
| `node_modules/` | 第三方依赖物理存放 | **依赖隔离（Dependency Isolation）** |
| `public/` | 编译产物输出 | **构建产物（Build Artifacts）** |

## 6. 出现问题

| 问题 | 现象 | 原因 | 解决方案 |
|:--|:--|:--|:--|
| `'node'` 不是内部或外部命令 | 终端无法识别 `node` / `npm` | 未勾选 *Add to PATH*，或未重启终端 | ① 重启终端；② 重装勾选 PATH；③ 手动将 `C:\Program Files\nodejs\` 加入系统环境变量 |
| `hexo init` 卡在 *Installing dependencies* | 长时间无响应，最终超时 | npm 默认源在海外，校园网 RTT > 200ms | `npm config set registry https://registry.npmmirror.com` 切换国内镜像 |
| `git clone` 主题极慢或失败 | 几十 KB/s 或 Connection refused | GitHub 在国内访问不稳定 | ① 走代理；② fork 到 Gitee；③ 直接下载 ZIP 解压到 `themes/` |
| `hexo deploy` 认证失败 | `remote: Invalid username or password` | GitHub 已禁用密码认证 | 使用 Personal Access Token 代替密码，或配置 SSH 密钥一键免密 |
| Pages 显示 404 | 浏览器打开 `用户名.github.io` 出 404 | ① 仓库名格式不对；② Pages 分支没选对 | ① 检查仓库名；② Settings → Pages → Branch 选 `master` |

### 7.5 连接 Git 版本管理

借助 **Git** 对 Hexo 博客项目和实验报告进行版本控制，实现代码与文档的**可追溯管理**。

**Git** 是目前全球最流行的分布式版本控制系统，它提供三项核心能力：

> 📸 **记录每一次修改** —— 哪怕只改了一个标点，也能精确存档
> ⏪ **随时读档回退** —— 让项目回到过去的任意存档点
> 🤝 **支持团队协作** —— 多人同时开发，智能合并代码

#### Git 的三个核心区域

```mermaid
flowchart LR
    Working["📝 工作区<br/>编辑中的文件<br/>（你看到的）"] -->|"git add"| Stage["📦 暂存区<br/>待提交的修改<br/>（购物车）"]
    Stage -->|"git commit"| Repo["🏛️ 仓库<br/>永久版本快照<br/>（档案馆）"]
```

| 区域 | 含义 | 类比 |
|:--:|:--:|:--:|
| 工作区（Working Directory） | 在 VS Code 中能看到并直接编辑的文件 | 书桌 —— 正在写的内容 |
| 暂存区（Staging Area） | 确认无误、准备提交的修改集合 | 购物车 —— 挑好待结账的 |
| 仓库（Repository） | 所有文件及完整修改历史的存储 | 档案馆 —— 永久保存 |

Git 最核心的工作循环只有三步：**修改 → 暂存 → 提交**。

#### 远程仓库：在云端管理代码

本地仓库解决了个人代码管理问题，但代码只存在本机 = 单点故障。**GitHub**（或国内访问更快的 **Gitee**）提供云端托管：

| 操作 | 含义 | VS Code 入口 | CLI 等价 |
|:--|:--|:--|:--|
| 推送（Push） | 将本地提交上传到远程 | `...` → 推送 | `git push` |
| 拉取（Pull） | 将远程更新下载到本地 | `...` → 拉取 | `git pull` |
| 克隆（Clone） | 将远程仓库完整下载到本地 | 源代码管理 → 克隆仓库 | `git clone <url>` |

> 📸 Git 的每一次 commit 都是项目历史的快照。回望时，能看到整个博客从零到上线的完整轨迹 —— 这就是版本控制的真正价值。

##　8. 心得体会

###　技术层面

这次实验让我第一次完整地走了一遍「从裸机到可运行网站」的 Web 开发环境搭建流程。最大的收获不是学会了敲 `hexo init` 这条命令，而是理解了这套工具链的**层次依赖关系**。

Node.js 是整个生态的「土壤」—— 没有它，npm、Hexo、以及成千上万的 npm 包都无法运行。npm 是「物流系统」—— 决定你用什么版本、从哪里下载、如何管理依赖树。Hexo 是「工厂流水线」—— 读取 Markdown 和主题模板，吐出静态 HTML。三者各司其职又环环相扣，与操作系统中的 *内核 → 包管理器 → 应用程序* 递进结构如出一辙。

另一个触动是 `hexo init` 阶段因镜像源卡了十分钟，让我切身体会到**网络基础设施的差异如何放大开发效率**。一段在国外 30 秒跑完的 `npm install`，在国内因 150+ 依赖、RTT 高、丢包重传而扩至 5–10 分钟。理解了这一点，再看 CI/CD 中「缓存 node_modules」「切换国内镜像」这些看似 trivial 的优化，就有了切身的量级概念。

###　思政层面

本次实验使用的 VS Code、Node.js、Git、Hexo 均为**开源软件**。如果没有这些全球开发者无偿贡献的工具，一个普通学生搭建个人博客的门槛会有多高？商业 IDE 授权费、专有构建系统的学习曲线、为部署一个静态页面而租用特定后端的服务器 —— 答案显而易见。

开源不只是「免费」—— 它让知识的获取变得更加平等。一个肯尼亚的学生和一个硅谷的学生，面对的是同一份 VS Code 源码、同一个 npm registry、同一条 `hexo init` 命令。这种**技术民主化**的力量，是开源运动最深刻的社会价值。

同时也注意到，当前 Web 工具链中具备统治地位的软件（Node.js、npm、Git、VS Code）几乎全部来自美国。npm registry 的核心服务器在海外，镜像只是「被动缓存」。如果极端情况下 npm 官方源对国内断连，大量项目的 `npm install` 将直接瘫痪 —— 2020 年 *left-pad 事件*中，一位开发者主动删包便引发数千项目构建失败，十几分钟内全球告警。技术主权的意义，在于「即便外部基础设施不可靠，核心工具链仍可运转」。国内虽有 cnpm、taobao mirror 等方案，本质仍是「缓存」而非「替代」。关注和推动真正的自主生态建设，是每一个技术从业者应有的格局。
