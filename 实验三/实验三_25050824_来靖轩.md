### 步骤五 配置静态 IP 地址

登录虚拟机后，编辑 Netplan 网络配置文件：

```bash
sudo nano /etc/netplan/50-cloud-init.yaml
```

修改为以下内容（MAC 地址用 `ip link show` 看到的真实值替换）：

```yaml
network:
  version: 2
  renderer: networkd
  ethernets:
    nat:
      match:
        macaddress: "08:00:27:0c:5d:66"           # NAT 网卡的 MAC
      dhcp4: yes
    lan:
      match:
        macaddress: "08:00:27:ec:42:e4"           # Host-Only 网卡的 MAC
      dhcp4: no
      addresses:
        - 192.168.56.101/24
      nameservers:
        addresses: [8.8.8.8, 114.114.114.114]
```

| 字段 | 含义 |
|:--:|:--:|
| `nat` / `lan` | 顶层标签，只是标识，可以随意取名 |
| `match: {macaddress: ...}` | 用 MAC 地址匹配网卡（绝对唯一，避免 altname 导致匹配失败） |
| `dhcp4: yes` | NAT 网卡自动获取 IP，无需手动管理 |
| `dhcp4: no` + `addresses` | Host-Only 网卡手动指定静态 IP `/24` |
| `nameservers` | DNS 服务器（Google `8.8.8.8` + 国内 `114.114.114.114`） |

应用配置：

```bash
sudo chmod 600 /etc/netplan/50-cloud-init.yaml
sudo netplan apply
```

验证：

```bash
ip addr show enp0s8        # 应显示 192.168.56.101/24
ping -c 3 8.8.8.8           # 外网连通性
ping -c 3 192.168.56.1      # 宿主机连通性
```

### 步骤八 修改克隆机的主机名和 IP

对每台克隆机分别执行。以 node1 为例：

> 💡 模板机 IP 配好后，建议在宿主机 PowerShell / CMD 里 `ssh dev@192.168.56.101` 连上虚拟机操作，命令可以直接从报告里复制粘贴，省去在 VirtualBox 小窗口里手打。

**1. 登录后先看网卡的真实 MAC 地址：**

```bash
ip link show
```

输出示例（每台克隆机的 MAC 都不同，以你实际看到的为准）：

```
2: enp0s9: <...> state DOWN ... link/ether 08:00:27:xx:xx:xx ...
    altname enx080027xxxxxxxx
3: enp0s8: <...> state UP   ... link/ether 08:00:27:xx:xx:xx ...
    altname enx080027xxxxxxxx
```

记下两张网卡各自的 `link/ether` 后面的 MAC 地址：
- `state DOWN` 的是 NAT 网卡
- `state UP` 的是 Host-Only 网卡

---

**2. 修改主机名：**

```bash
sudo hostnamectl set-hostname node1
exit                          # 退出登录，重新 SSH 连入即可看到前缀变为 node1
```

---

**3. 清理可能残留的多余 netplan 配置文件：**

```bash
ls /etc/netplan/
```

如果存在多个 `.yaml` 文件（如 `00-installer-config.yaml`），只保留一个，其余删掉：

```bash
sudo rm /etc/netplan/00-installer-config.yaml    # 示例，以实际看到的为准
```

> netplan 会合并该目录下所有 yaml 文件，残留的旧配置可能仍引用失效的接口名导致匹配报错。

---

**4. 编辑 netplan，用 MAC 地址匹配（彻底避开 altname 问题）：**

```bash
sudo nano /etc/netplan/50-cloud-init.yaml
```

```yaml
network:
  version: 2
  renderer: networkd
  ethernets:
    nat:
      match:
        macaddress: "08:00:27:xx:xx:xx"       # NAT 网卡的 MAC（state DOWN 那张）
      dhcp4: yes
    lan:
      match:
        macaddress: "08:00:27:xx:xx:xx"       # Host-Only 网卡的 MAC（state UP 那张）
      dhcp4: no
      addresses:
        - 192.168.56.102/24     # node2→103，node3→104
      nameservers:
        addresses: [8.8.8.8, 114.114.114.114]
```

> 顶层键名 `nat` / `lan` 只是标签，可以随便取名。用了 `match: {macaddress: ...}` 之后 netplan 只看 MAC 地址，不管接口名是 enp0s8 还是 enp0s9 还是有什么 altname，通通没问题。

---

**5. 应用配置：**

```bash
sudo chmod 600 /etc/netplan/50-cloud-init.yaml
sudo netplan apply
```

---

**6. 更新 /etc/hosts，确保四台机器都有主机名解析：**

```bash
sudo nano /etc/hosts
```

在末尾追加（如果已有则跳过）：

```
192.168.56.101  template
192.168.56.102  node1
192.168.56.103  node2
192.168.56.104  node3
```

---

**7. 验证：**

```bash
ip addr                           # 确认 Host-Only 网卡拿到了 192.168.56.102/24
ping -c 3 8.8.8.8                 # 外网（确认 NAT 网卡没丢）
```

node2、node3 照搬上述流程，区别只有两处（见下表）：

| 节点 | 主机名 | Host-Only IP | netplan 中改这一行 |
|:--:|:--:|:--:|:--:|
| 模板机 | template | 192.168.56.101 | （不克隆，保持不变） |
| 克隆 1 | node1 | 192.168.56.102 | `192.168.56.102/24` |
| 克隆 2 | node2 | 192.168.56.103 | `192.168.56.103/24` |
| 克隆 3 | node3 | 192.168.56.104 | `192.168.56.104/24` |

> ⚠️ 每台克隆机的 MAC 地址都不同，**不要复制模板机的 MAC**。先在每台上跑 `ip link show`，用实际看到的 MAC 地址填入配置。

### 步骤九　验证多机环境

在 node1 上执行四组 ping 测试：

```bash
ping -c 3 8.8.8.8                  # 外网连通
ping -c 3 192.168.56.1             # 宿主机连通
ping -c 3 node2                    # 跨虚拟机连通（验证 hosts 解析）

    style HostOnly fill:#e8f5e9,stroke:#2e7d32
```

本实验用 **NAT + Host-Only 双网卡** 的原因：NAT 负责 `apt install` 等外网操作，Host-Only 提供固定 IP 的私网互通 —— 两者各司其职，互不干扰。

---

### 5.3 克隆 vs 快照

| 操作 | 产物 | 是否独立 | 适用场景 |
|:--:|:--:|:--:|:--:|
| **完全克隆** | 完整独立的虚拟硬盘副本 | 是 | 批量创建多台节点 |
| **链接克隆** | 共享原始磁盘，只存储差异 | 否（依赖母盘） | 节省空间但不可迁移 |
| **快照** | 保存虚拟机状态，可随时回滚 | 否（依赖原机） | 实验前留回退点 |

## 6. 出现问题

| 问题 | 现象 | 原因 | 解决方案 |
|:--:|:--:|:--:|:--:|
| `netplan apply` 报 Cannot find unique interface | 执行 `sudo netplan apply` 报错找不到接口，但 `ip link show` 能看到对应接口名且名字正确 | 网卡存在 `altname`（如 `enx08002732c107`）：同一个接口同时有 `enp0s8` 和 `enx…` 两个名字，netplan 仅凭名字自动匹配时发现"两个名字指向同一设备"，判定不唯一。克隆后 MAC 变更更会加剧此问题 | 改用 `match: {macaddress: "xx:xx:xx:xx:xx:xx"}` 以 MAC 地址匹配（绝对唯一）。先在每台机器上 `ip link show` 获取真实 MAC，填入 netplan 配置 |
| `netplan apply` 配置正确但仍报接口不唯一 | netplan 配置已改为 MAC 匹配格式，但依旧报 `Cannot find unique matching interface` | `/etc/netplan/` 下存在多个 `.yaml` 文件（如 `00-installer-config.yaml` + `50-cloud-init.yaml`），netplan 会合并所有文件，残留的旧配置仍在引用接口名 | `ls /etc/netplan/` 查看，`sudo rm` 删掉多余的 yaml 文件（只保留一个），再 `sudo netplan apply` |
| `ssh: connect to host port 22: Connection refused` | 无法 SSH 登录虚拟机 | 安装时未勾选 *Install OpenSSH server* | 在虚拟机中执行 `sudo apt update && sudo apt install openssh-server -y` |
| 虚拟机 ping 宿主机不通 | `ping 192.168.56.1` 超时，但宿主机可以 ping 通虚拟机 | Windows 防火墙默认拦截来自 Host-Only 网卡的入站 ICMP 包 | **临时方案**：Windows 防火墙 → 高级设置 → 入站规则 → 启用"文件和打印机共享(回显请求 - ICMPv4-In)"——仅限专用网络；或直接关闭防火墙测试。**永久方案**：新建入站规则 → 协议 ICMPv4 → 远程 IP 设为 `192.168.56.0/24` |

## 7. 拓展与进阶

### 7.1 快照管理

实验前留快照，操作失误一键回滚：

```bash
# VirtualBox 图形界面
选中 VM → 右上角 "快照" → "生成" → 命名（如"A-初始配置"）

# 恢复快照
选中快照 → "恢复备份"

# CLI 方式
VBoxManage snapshot template take "before-experiment"
VBoxManage snapshot template restore "before-experiment"
```

---

### 7.2 端口转发（从宿主机 SSH 到 VM）

NAT 模式下外界无法直接访问 VM。通过端口转发，将宿主机端口映射到虚拟机 SSH：

```bash
VBoxManage modifyvm "template" --natpf1 "ssh,tcp,,2222,,22"
```

之后可在宿主机 PowerShell 中 `ssh dev@localhost -p 2222` 直连虚拟机 —— 无需切换到 Host-Only 网络，适合单机开发。

## 8. 心得体会

装系统这件事本身不新鲜，但用虚拟化方式装，感受完全不一样。以前重装系统要腾 U 盘、进 BIOS 改启动顺序、覆盖磁盘分区 —— 每一步都怕出错。VirtualBox 打开之后点几下鼠标就能起一台全新的 Ubuntu，装毁了也不怕，删掉重来就是。这种"零成本试错"的快感，让人更愿意动手尝试，而不是看教程畏手畏脚。

四台机器互 ping 全通的那一刻，我突然理解了为什么运维和分布式系统课程都从虚拟机上手 —— 在这个虚拟私网里，可以做任何你想做的实验：搭集群、配负载均衡、模拟网络分区故障。而这一切只占用了你宿主机上一块 80GB 的硬盘空间。虚拟化技术本质上就是把昂贵的硬件资源「切分」和「隔离」，让每个人都能拥有一片属于自己的机房。这种将物理资源抽象为软件接口的思想，从云计算到容器再到 K8s，贯穿了整个后端基础设施的演进史。
