*△ 图 2 · VS Code Markdown All in One插件*

| 功能 | 快捷键 | 效果 |
|:--|:--:|:--:|

`Ctrl+,` 打开设置界面，修改以下三项：

| 设置项 | 值 | 效果 |

### 步骤四 Markdown 语法全解

`Ctrl+N` → `Ctrl+S` 保存为 `test.md`。以下逐一覆盖 Markdown 从基础到高级的全部常用语法 —— 每一类语法都可直接在 VS Code 右侧预览窗口实时查看效果。

#### 4.1 标题与段落

Markdown 提供六级标题（对应 HTML `<h1>` ~ `<h6>`），用 `#` 数量控制层级：

```markdown
# 一级标题
## 二级标题
### 三级标题
#### 四级标题
##### 五级标题
###### 六级标题
```
![alt text](image-7.png)
*△ 图 6 · 标题与段落渲染*

段落直接书写，中间用一个**空行**分隔。若只换行而不空行，渲染器不会另起段落 —— 这一规则初学时常踩坑：

```markdown
这是第一段。

这是第二段（中间有空行，会分段）。
这是同一段的续行（中间只换行，不会分段）。
```

> 💡 若想在段内强制换行，在行末加**两个空格**再回车。

---

#### 4.2 文本样式

| 语法 | 渲染效果 | 使用场景 |
|:--|:--|:--|
| `**粗体**` 或 `__粗体__` | **粗体文本** | 强调关键概念、术语 |
| `*斜体*` 或 `_斜体_` | *斜体文本* | 书名、作品名、引用短语 |
| `***粗斜体***` | ***粗斜体*** | 极强调 |
| `~~删除线~~` | ~~删除线~~ | 表示已废弃或勘误 |
| `==高亮==` | ==高亮标记== | 重点突出（部分渲染器支持） |
| `<u>下划线</u>` | <u>下划线文本</u> | Markdown 无原生下划线，用 HTML 实现 |
| `H~2~O` / `x^2^` | H~2~O / x² | 上下标（部分渲染器支持） |

#### 4.10　LaTeX 数学公式

#### 4.12　HTML 混编

Markdown 文件中可以直接嵌入 HTML 标签，适合实现 Markdown 原生不支持的排版效果：

```markdown
<details>
<summary>点击展开详情</summary>

这里是被折叠的内容，里面可以有 **Markdown** 语法。

```python
print("代码块也可以放在折叠区内")
```

</details>

<p align="center">这段文字会居中显示</p>

<kbd>Ctrl</kbd> + <kbd>S</kbd> 保存文件
```

### 步骤五 Git 版本管理

在讲解 Markdown 语法的过程中，我们已经创建了 `test.md` 等多个文件。如何锁定每一次修改的快照、如何在误删之后找回历史版本 —— 这正是 **Git** 要解决的问题。

#### 为什么需要版本控制

设想你在写实验报告，文件夹变成了这样：

```
实验报告_v1.md
实验报告_v2_修改了表格.md
实验报告_final.md
实验报告_final_真最终版.md
```

哪一份才是最新的？「真最终版」到底改了什么？靠文件命名管理版本，混乱且不可回溯 —— 这就是**版本控制系统（VCS）** 存在的理由。

**Git** 是目前全球最流行的分布式版本控制系统，提供三项核心能力：

> 📸 **记录每一次修改** —— 哪怕只改了一个标点，也能精确存档
> ⏪ **随时读档回退** —— 让项目回到过去的任意存档点
> 🤝 **支持团队协作** —— 多人同时开发，智能合并代码

---

#### Git 的三个核心区域

```mermaid
flowchart LR
    Working["📝 工作区<br/>编辑中的文件<br/>（你看到的）"] -->|"git add"| Stage["📦 暂存区<br/>待提交的修改<br/>（购物车）"]
    Stage -->|"git commit"| Repo["🏛️ 仓库<br/>永久版本快照<br/>（档案馆）"]

    style Working fill:#fff3e0,stroke:#f57c00
    style Stage fill:#e8f5e9,stroke:#2e7d32
    style Repo fill:#e3f2fd,stroke:#1565c0
```

| 区域 | 含义 | 类比 |
|:--:|:--:|:--:|
| 工作区（Working Directory） | 在 VS Code 中能看到并直接编辑的文件 | 书桌 —— 正在写的内容 |
| 暂存区（Staging Area） | 确认无误、准备提交的修改集合 | 购物车 —— 挑好待结账的 |
| 仓库（Repository） | 所有文件及完整修改历史的存储 | 档案馆 —— 永久保存 |

Git 最核心的工作循环只有三步：**修改 → 暂存 → 提交**。

---

#### 实操：将本实验项目纳入 Git 管理

**▸ 第一步：暂存文件**

VS Code left sidebar → **源代码管理** 图标（`Ctrl+Shift+G`）。所有未追踪的文件出现在 *"更改"* 列表下，标记为 `U`（Untracked）。鼠标悬停到文件上，点击右侧出现的 **`+`** 号 —— 文件移至 *"已暂存的更改"*，状态变为 `A`（Added）。

![暂存文件操作1](image-27.png)
![暂存文件操作2](image-28.png)
*△ 图 32 图 33 · 暂存文件操作。左侧原始状态（U），右侧点击 + 号后移至暂存区*

```bash
git add .    # 等价 CLI 操作
```

**▸ 第二步：提交**

在面板顶部的 *"消息"* 输入框中填写提交说明，点击上方的 **`✓`**（提交）按钮。*"更改"* 和 *"已暂存的更改"* 列表变空 —— 说明工作区与仓库当前版本已完全同步。

![提交操作](image-29.png)
*△ 图 34 · 提交操作 —— 输入提交信息后点击 ✓*

```bash
git commit -m "feat: 完成 Markdown 语法练习"    # 等价 CLI 操作
```

**▸ 第三步：修改后再次提交**

修改任意文件并保存 → 文件出现在更改列表中，标记为 `M`（Modified）→ 点击 `+` 暂存 → 输入说明 → 点击 `✓` 提交。

![修改后提交1](image-30.png)
![修改后提交2](image-31.png)
*△ 图 35 36 · 修改文件后，文件状态变为 M（Modified），再次执行暂存 → 提交*

```bash
git add -A
git commit -m "docs: 补充 LaTeX 公式示例"
```

> 🔁 这组操作 —— **修改 → 暂存 → 提交** —— 是本实验以及日后开发中频率最高的工作流。养成「每完成一个小目标就 commit 一次」的习惯，历史记录会干净且可回退。

**提交信息规范建议：**

| 前缀 | 含义 | 示例 |
|:--:|:--:|:--:|
| `feat:` | 新增功能 / 内容 | `feat: 添加 Mermaid 图表教程` |
| `fix:` | 修复错误 | `fix: 修复表格对齐` |
| `docs:` | 文档更新 | `docs: 补全 LaTeX 符号表` |
| `chore:` | 杂项维护 | `chore: 整理截图文件` |

#### 远程仓库：在云端管理代码

本地仓库解决了个人版本管理问题，但代码只存在本机 = 单点故障。**GitHub**（或国内访问更快的 **Gitee**）提供云端托管：

| 操作 | 含义 | VS Code 入口 | CLI 等价 |
|:--:|:--:|:--:|:--:|
| 推送（Push） | 将本地提交上传到远程 | `...` → 推送 | `git push` |
| 拉取（Pull） | 将远程更新下载到本地 | `...` → 拉取 | `git pull` |
| 克隆（Clone） | 将远程仓库完整下载到本地 | 源代码管理 → 克隆仓库 | `git clone <url>` |

> 📸 Git 的每一次 commit 都是项目历史的快照。回望时，能看到整个实验从零到完成的完整轨迹 —— 这就是版本控制的真正价值。

## 6. 出现问题

| 问题 | 现象 | 原因 | 解决方案 |
|:--:|:--:|:--:|:--:|
| `code` 命令无效 | 终端输入 `code .` 提示「不是内部或外部命令」 | VS Code 安装时未勾选 *添加到 PATH* | 重装勾选 PATH，或手动将 `C:\Users\用户名\AppData\Local\Programs\Microsoft VS Code\bin` 加入环境变量 |
| 预览窗口渲染异常 | `Ctrl+K V` 打开预览后空白或乱码 | Markdown Preview Enhanced 未正确激活 | 在扩展面板确认插件已启用；或右键 `.md` 文件 → *打开方式* → 选择 VS Code |
| 图片粘贴后路径引用失效 | `Ctrl+Alt+V` 粘贴图片，预览中显示裂图 | `Paste Image: Path` 设为绝对路径，换电脑后路径不存在 | 改为 `${currentFileDir}/images/` 相对路径 |
| `[TOC]` 不生成目录 | 文件保存后 `[TOC]` 仍是原样 | 未安装 Markdown All in One 或其未启用 | 安装并启用后，`Ctrl+Shift+P` → *Create Table of Contents* 手动触发 |
| 扩展面板搜索不到插件 | 输入关键词后一直转圈 | ① 网络问题；② 搜索关键词不准确 | 检查网络；使用插件全名（如 `Markdown All in One`）搜索 |
| markdownlint 误报干扰 | 波浪线提示过多，影响写作 | 默认启用了大量语法规则，部分与实际需求冲突 | `.markdownlint.json` 中按需禁用特定规则，例如 `"MD013": false`（关闭行宽限制） |

## 7. 心得体会

实验二和实验一其实是互为表里的关系。实验一搭了一条从 Markdown 到 HTML 再到公网的流水线，实验二解决的则是流水线最上游的问题 —— 内容怎么写才高效。VS Code 这个工具本身不复杂，但把四个插件（All in One、Preview Enhanced、Paste Image、markdownlint）串起来之后，写作体验的变化是立竿见影的：左边敲源码，右边看渲染效果，截图 `Ctrl+Alt+V` 直接进文档，保存后 `[TOC]` 自动刷成目录 —— 以前用 Word 打实验报告，排版调字号比写内容还费时间，现在一边写一边看效果，思路完全不会被打断。

这套工具链最打动我的地方在于 —— VS Code 免费、四个插件全部开源、Markdown 本身是一个开放标准。也就是说，任何一个连上网的人，理论上都能零成本获得一套不输商业写作工具的编辑环境。工具的门槛降得越低，能参与创作的人就越多。我自己写完那份实验一报告，就是从 `test.md` 里练出来的语法一样一样往上拼的，最后再导出 PDF 交作业，全程没离开过 VS Code。这个流畅感让人觉得，学这些工具是值得的。
