**1.2 理解 Shell —— 你和内核之间的翻译官**

Shell（壳）是裹在 Linux 内核外面的一层命令解释器。你敲的命令通过 Shell 翻译成系统调用（system call），再由内核去操作硬件。常见的 Shell：

| Shell | 提示符 | 特点 |
|:--:|:--:|:--:|
| bash | `$` | Linux 标配，使用最广 |
| zsh | `$` | 兼容 bash，主题/插件丰富 |
| sh | `$` | 最原始、最轻量的 POSIX Shell |

`echo $SHELL` 可查看当前使用的 Shell。

```bash
echo $SHELL                         # 当前 Shell（输出 /bin/bash）
```

**2.2 看看周围：`ls`**

`ls`（List）是最常敲的命令，没有之一：

```bash
ls                                  # 列出当前目录下的非隐藏文件
ls -l                               # long format：权限 · 硬链接数 · 所有者 · 组 · 大小 · 修改时间 · 文件名
ls -la                              # 含隐藏文件（文件/目录名以 . 开头即为隐藏）
ls -lh                              # human-readable：文件大小 32K / 4.0M 而不是 32768 / 4194304
ls -ltr                             # 按修改时间排序（t），倒序（r），最近改的在最下面
```

`ls -l` 输出的每一列含义：

```
-rw-r--r--  1  dev  dev   4096  Jul 10 10:00  hello.txt
|           |  |    |     |      |            |
权限字串    硬链接 所有者 组   文件大小  最后修改时间  文件名
```

权限字串 `-rw-r--r--` 拆解（共 10 位）：

| 位 | 含义 | 取值 |
|:--:|:--:|:--:|
| 1 | 文件类型 | `-` 普通文件 / `d` 目录 / `l` 符号链接 |
| 2-4 | 所有者权限 | `r` 读 `w` 写 `x` 执行 |
| 5-7 | 同组用户权限 | 同上 |
| 8-10 | 其他人权限 | 同上 |

**2.4 Linux 根目录下每个文件夹是干嘛的？**

| 目录 | 全称 | 存了什么 |
|:--:|:--:|:--:|
| `/bin` | Binaries | 系统基础命令（`ls`、`cp`、`cat` 等） |
| `/boot` | Boot | 内核镜像、引导加载器 |
| `/dev` | Devices | 设备文件（磁盘、终端、鼠标等，一切皆文件） |
| `/etc` | Editable Text Config | 系统级配置文件（网络、SSH、用户密码等） |
| `/home` | Home | 每个普通用户的个人目录（`/home/dev`） |
| `/lib` | Libraries | 系统库文件（`.so`，类似 Windows 的 `.dll`） |
| `/mnt` | Mount | 临时挂载点（U 盘、共享文件夹） |
| `/opt` | Optional | 第三方商业软件（如 VMware Tools） |
| `/proc` | Process | 虚拟文件系统，存进程和内核运行信息 |
| `/root` | Root | root 超级用户的主目录 |
| `/tmp` | Temporary | 临时文件，重启后自动清空 |
| `/usr` | Unix System Resources | 用户级程序、库、文档（最大的目录之一） |
| `/var` | Variable | 日志、缓存、数据库等可变数据 |

安装 `tree` 以树形图直观感受目录层级：

### 步骤四 搜索与检索 —— 大海捞针

**4.1 `grep`：从文件内容中搜索**

`grep`（Global Regular Expression Print）是文本搜索的事实标准，日常使用频率极高。

```bash
grep "dev" /etc/passwd              # 在文件中搜索含 "dev" 的行
grep -r "netplan" /etc/             # 递归搜索 /etc/ 下所有文件
grep -n "root" /etc/passwd          # 显示行号
grep -v "nologin" /etc/passwd       # 反向匹配：排除含 "nologin" 的行
grep -i "Error" /var/log/syslog     # 忽略大小写
grep -c "ssh" /var/log/syslog       # 只统计匹配行数
```

`/etc/passwd` 每行格式：
```
dev:x:1000:1000:dev:/home/dev:/bin/bash
 |   |   |    |    |        |        └─ Shell
 |   |   |    |    |        └─ 主目录
 |   |   |    |    └─ 用户全名（GECOS）
 |   |   |    └─ 用户组 ID（GID）
 |   |   └─ 用户 ID（UID 1000+ = 普通用户，0 = root）
 |   └─ 密码占位符（真实密码在 /etc/shadow）
 └─ 用户名
```

### 步骤八 管道与重定向 —— 命令行的灵魂

管道和重定向是 Linux 命令行最强大的组合技。如果说前面的命令是"单词"，管道和重定向就是"语法"——学会了它们，你才能说出一句完整的 Linux 句子。


**8.1 理解 stdin / stdout / stderr**

Linux 下每个程序都有三条"默认通道"：

| 通道 | 编号 | 全称 | 默认流向 | 比喻 |
|:--:|:--:|:--:|:--:|:--:|
| 标准输入 | 0 | stdin | 键盘 | 嘴巴 —— 程序从这里"吃"数据 |
| 标准输出 | 1 | stdout | 终端屏幕 | 正常说话 —— 程序往这里输出结果 |
| 标准错误 | 2 | stderr | 终端屏幕 | 咳嗽/骂人 —— 错误信息也走屏幕，但走不同通道 |

理解这三个通道是掌握重定向的前提。

```bash
# 正常输出走 stdout（通道1）
echo "这是一条正常消息"

# 错误走 stderr（通道2），需要故意触发一个错误
ls /nonexistent
# ls: cannot access '/nonexistent': No such file or directory
```

![alt text](image-30.png)
*△ 图 29 · `stdout` 正常输出 vs `stderr` 错误输出 —— 两条通道一目了然*

**8.2 什么是管道**

管道符 `|` 把左边命令的 stdout 直接接到右边命令的 stdin。数据从左到右，像流水线一样传递。

> 💡 **补充：管道 `|` 的原理是什么？**
> 答：Shell 创建一条匿名管道（pipe）——内核中的一块环形缓冲区。左边进程往管道的写端写数据，右边进程从管道的读端读数据。当一个命令的输出速率远快于另一个的输入速率时，内核会自动"刹车"（阻塞），保证数据不丢失。

用 `lab/logs/error.log`（之前建的实验日志）来跑两条实例：

```bash
# 示例1：列出 /etc 下所有文件，只保留含 "net" 的
ls /etc | grep net

# 示例2：统计 lab 目录下有多少条目（含隐藏文件）
ls -la ~/lab | wc -l

# 示例3：查看 error.log 里所有 ERROR 日志
cat ~/lab/logs/error.log | grep ERROR
```

![alt text](image-31.png)
*△ 图 30 · 管道三连：`ls` + `grep`、`ls` + `wc`、`cat` + `grep`*

继续进阶——管道可以级联任意长度：

```bash
# 三级管道：列出所有文件 → 过滤含"log"的 → 统计行数
ls -la ~/lab | grep log | wc -l

# 从进程列表找到 SSH
ps aux | grep ssh | head -5
```

> 💡 **补充：管道 `|` 和 `>` 有什么区别？**
> 答：`|` 把输出传给**另一个命令**；`>` 把输出存到**文件**。一个流向程序，一个流向磁盘。

*△ 图 31 · 三级管道：`ls` → `grep` → `wc`，数据经过两次传递*

**8.3 输出重定向 `>` 和 `>>`**

管道把数据喂给程序，重定向把数据存到文件：

```bash
cd ~
# 单 > 覆盖写入（文件原有内容会被清空）
ls -la ~/lab > snapshot.txt
cat snapshot.txt

# 双 >> 追加写入（在文件末尾续写，不覆盖）
echo "=== 快照时间: $(date) ===" >> snapshot.txt
cat snapshot.txt
```

> 💡 **补充：`>` 和 `>>` 的区别？**
> 答：`>` 覆盖写入（先清空文件再写），`>>` 追加写入（保留旧内容，在末尾续写）。

![alt text](image-33.png)
*△ 图 32 · 输出重定向 —— `>`覆盖 vs `>>`追加*

**8.4 输入重定向 `<`**

大多数命令可以从文件读取，不需要 `<`（如 `cat file` 本身就支持文件名参数）。但 `wc` 这类命令走 stdin 更自然：

```bash
# 两种等价写法
wc -l /etc/passwd                    # 传文件名
wc -l < /etc/passwd                  # 输入重定向（只输出行数，不输出文件名）
```

区别：传文件名时 `wc` 会同时打印文件名，输入重定向不会。

> 💡 **补充：`grep "foo" file` 和 `grep "foo" < file` 的区别？**
> 答：结果一样，但 `< file` 是 Shell 打开的，文件不存在时 Shell 会报错；传文件名是 grep 自己打开的，grep 报错。权限不足时行为也略有差异。

![alt text](image-34.png)
*△ 图 33 · 输入重定向 `<`——`wc -l` 传文件名 vs 重定向*

**8.5 stderr 重定向**

错误信息和正常输出走不同的通道，可以分开处理：

```bash
# 只重定向 stdout（错误仍然打印到屏幕）
ls /etc/netplan 1> stdout.txt
cat stdout.txt

# 只重定向 stderr（正常输出仍然打印到屏幕）
ls /nonexistent 2> stderr.txt
cat stderr.txt

# 把 stdout 和 stderr 都丢到同一个文件
ls /etc/netplan /nonexistent > both.txt 2>&1
cat both.txt
```

> 💡 **补充：解释 `command > file 2>&1` 这个写法的含义和顺序为什么不能颠倒。**
> 答：`> file` 先把 stdout 重定向到 file，`2>&1` 再把 stderr 重定向到"stdout 当前指向的地方"（也就是 file）。如果写成 `2>&1 > file`，stderr 会重定向到"旧的 stdout"（终端），只有 stdout 进 file —— 这就是常见的"错误还在终端打印"的坑。

![alt text](image-35.png)
*△ 图 34 · `stderr` 重定向 —— 分开捕获正常输出和错误输出*

**8.6 `/dev/null` —— 黑洞**

`/dev/null` 是一个特殊的"黑洞文件"：写进去的数据直接消失，读永远是空的。常用于**丢弃不需要的输出**：

```bash
# 只要结果（退出码），不要输出
ping -c 2 8.8.8.8 > /dev/null && echo "网络通" || echo "网络不通"

# 丢弃错误信息，只看正常结果
find /etc -name "hosts" 2>/dev/null
```

> 💡 **补充：`/dev/null` 是什么？有什么用途？**
> 答：是 Linux 的伪设备文件，写入其中的数据会被立即丢弃。常见用途：① 丢弃不需要的输出（静默执行）；② 清空文件内容（`cat /dev/null > file`）；③ 作为测试输入源（读取时永远返回 EOF）。

### 5.3 绝对路径 vs 相对路径

| 类型 | 写法 | 特点 |

### 5.4 APT 包管理器架构

```
apt（上层命令行）
   ↓ 调用
dpkg（底层 .deb 安装器）
   ↓ 安装
.deb 包（二进制软件包）
```

- `apt`：自动解析依赖，"你要 A 我就把 B 和 C 一起拉下来"
- `dpkg`：只装指定的包，不处理依赖
- `snap`：Ubuntu 力推的新一代包格式（自带依赖，沙箱隔离），本实验不涉及

## 6. 出现问题

| 问题 | 现象 | 原因 | 解决方案 |
|:--:|:--:|:--:|:--:|
| `tree: command not found` | 输入 `tree` 提示命令不存在 | 最小化安装不包含 tree | `sudo apt install tree -y` |
| `rm: cannot remove 'xxx': Is a directory` | `rm xxx` 报错 | 默认 `rm` 不能删目录 | 用 `rm -r xxx` 递归删除 |
| `Permission denied` | 操作 `/etc` 下文件时报权限拒绝 | 普通用户无权修改系统配置 | 加 `sudo` 临时提权 |
| `apt update` 极慢或超时 | `sudo apt update` 卡住不动 | Ubuntu 默认源在海外 | 参考实验一切换国内镜像：`sudo sed -i 's/archive.ubuntu.com/mirrors.ustc.edu.cn/g' /etc/apt/sources.list` |
| `grep` 匹配到不相关的行 | `grep ssh` 把 `ssh-agent` 也搜到了 | 关键词太短 | `grep -w "ssh"` 精确匹配整个词 |
| `mkdir` 父目录不存在 | `mkdir a/b/c` 报错 `No such file or directory` | 缺少 `-p` | `mkdir -p a/b/c`，-p 自动创建不存在的父目录 |

## 7. 心得体会

实验四的操作量不小，但命令按"导航 → 操作文件 → 搜索 → 装软件 → 看系统 → 管进程 → 玩管道"这条线走下来，逻辑很清楚。

最大的认知转变是：Linux 桌面其实可有可无，命令行才是本体。`ls -la` 看权限比 Windows 资源管理器那密密麻麻的属性页快得多，`grep -r "关键字" /etc/` 搜配置比任何全文搜索工具都快。一个鼠标都懒得够的服务器运维，所有操作都在这个黑色窗口里完成——我现在有点理解那种感觉了。

`|` 管道是我认为整个实验最漂亮的设计。它让命令行变成了一套"数据流水线"——你不需要中间变量、不需要临时文件、不需要写代码，就把一个个原子化的小命令像乐高积木一样搭成任意长度的处理链条。这种 KISS（Keep It Simple, Stupid）设计哲学是所有优秀工程实践的底层逻辑，不只是 Linux。

最后，大一能接触到这些确实是好事。等到大三数据结构课设、操作系统实验、甚至毕业设计部署服务的时候，别人在查"怎么在 Linux 上装 Python"，你已经在 `grep`、`chmod`、`ps aux` 的加持下写一键部署脚本了——这差距从一开始就拉开了。
